## Architecture Patterns

### Backend
- **Pattern**: [Controller тЖТ Service тЖТ Repository / Hexagonal (Ports & Adapters) / CQRS]
- **Layer responsibilities**:
  - Controller: parse and validate request, delegate to service, return response
  - Service: business logic, orchestration, domain rules
  - Repository: data access only тАФ no business logic, no HTTP concerns
- **Error handling**: [e.g., Result type / thrown exceptions / Problem Details RFC 9457]
- **SOLID notes**: [Any project-specific SOLID decisions, e.g., "Services expose interfaces тАФ concrete classes injected via DI container"]

### Frontend
- **Folder structure**: [Feature-first / Atomic Design]
  - Feature-first example: `features/auth/`, `features/dashboard/`, `shared/components/`
  - Atomic Design example: `atoms/`, `molecules/`, `organisms/`, `templates/`, `pages/`
- **Component model**: [Co-located (component + test + styles in same folder) / Centralized]
- **Design token usage**: Primitive tokens (`--blue-500`) тЖТ Semantic tokens (`--color-primary`) тЖТ Component tokens (`--button-bg`). Always use semantic tokens in components, never primitives directly.
- **SOLID notes**: [e.g., "All data fetching in hooks, never in component bodies. Extend via props/composition, never modifying existing components."]

### Smart Contracts *(if in scope)*
- **Upgradeability**: [Immutable / UUPS proxy / Transparent proxy / Diamond EIP-2535]
  - Rationale: [Why this choice]
- **Factory pattern**: [Yes тАФ deploy multiple instances via a factory contract / No]
- **Registry pattern**: [Yes тАФ contracts discover each other via a registry / No]
- **Payment disbursement**: Pull payment pattern (users withdraw) тАФ never push ETH to arbitrary addresses
- **SOLID notes**: [e.g., "Each contract has one responsibility. Cross-contract deps injected via constructor. Interface segregation: one slim interface per consumer role."]
