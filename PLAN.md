### Information hierarchy principle

Each piece of information lives in exactly one place:
- **Mode definitions** live in `docs/workflow/modes.md` — README and help skill link to it, never duplicate it
- **Handoff templates** live in `docs/workflow/handoff-templates.md` — skills reference it, never inline the templates again
- **Gate criteria** live at the bottom of each phase doc — co-located with the phase they govern

## Content Contracts

What each new or modified file must contain. These are the acceptance criteria for each doc task.

### NEW: `docs/workflow/handoff-templates.md`

Four fenced markdown templates (copy-pasteable into agent prompts):

**Template 1 — Task Assignment** (used by `/build` when spawning an agent)
- Fields: `To` (agent role), `Task` (ID + description from task.md), `Context` (relevant PLAN.md section, files to read, acceptance criteria), `Scope` (do not modify outside these files), `Handoff to` (who receives the output)

**Template 2 — QA PASS** (used by qa-runner in strict mode)
- Fields: `Task`, `Attempt N of 3`, `Verdict: PASS`, `Evidence` (tests run, criteria checked), `Next action` (mark [x] in task.md, advance)

**Template 3 — QA FAIL** (used by qa-runner in strict mode)
- Fields: `Task`, `Attempt N of 3`, `Verdict: FAIL`, `Issues` (each with: Expected / Actual / File:line / Fix instruction), `Acceptance criteria status` ([x] / [ ] per criterion), `Retry instructions` (fix only listed issues, no new features, re-run /check first)

**Template 4 — Escalation** (used when 3 retries exhausted)
- Fields: `Task`, `Attempts: 3/3`, `Failure history` (what was tried + why it failed for each attempt), `Root cause`, `Resolution options` (decompose / revise approach / defer / accept with limitation), `Impact` (what other tasks are blocked, what quality compromise exists if accepted)

### NEW: `docs/workflow/modes.md`

Single-page reference. Three sections:

**Full mode** — when: new project from scratch / phases: all 6 / skills: `/plan` → `/build` → `/validate` → `/ship` / note: the default README quickstart is the Full walkthrough

**Sprint mode** — when: adding a feature to an existing project (SPEC and design may already exist or be minimal) / phases: 3–6 / skills: `/new-feature` → `/plan` → `/build` → `/validate` → `/ship` / note: skip phases 1–2 only if SPEC.md already exists for the feature area

**Micro mode** — when: single targeted change (one bug fix, one component, one route, one test) / phases: none / skills: `/new-feature` → `gen-component` | `gen-api-route` | `add-test` → `/check` → `/ship` / note: skip planning entirely; acceptable only for changes that touch ≤3 files

Plus a one-table decision guide at top: Mode | Use when | Time estimate

Append a gate section at the end:

**Exit Gate — Phase 3 → 4**

| Criterion | Threshold | How to verify |
|---|---|---|
| All TBDs resolved in CLAUDE.md files | 0 remaining `TBD` values | `grep -r "TBD" CLAUDE.md backend/CLAUDE.md frontend/CLAUDE.md contracts/CLAUDE.md` |

| Human has approved PLAN.md | Verbal or written approval | Human sign-off before Phase 4 begins |

Gate fails → revise PLAN.md, re-run relevant parts of `/plan`, repeat.

### MODIFIED: `docs/workflow/phase4-build.md`

Append a gate section at the end:

**Exit Gate — Phase 4 → 5**

| Criterion | Threshold | How to verify |
|---|---|---|
| All tasks complete | 0 unchecked `- [ ]` lines | `grep -c "\- \[ \]" task.md` returns 0 |

Append a gate section at the end:

**Exit Gate — Phase 6 (Ready to Merge)**

| Criterion | Threshold | How to verify |
|---|---|---|
| `/check` passes | 0 failures | Run `/check` immediately before committing |
| Branch rebased on latest `main` | 0 commits behind | `git status` shows up to date with origin/main |
| PR title ≤ 70 characters | ≤ 70 chars | Count characters in PR title |
| PR body has: summary, test plan, SPEC.md story link | All three sections present | Manual review of PR body |
| All CI checks green | 0 failing checks | GitHub CI status page |
| Human approval received | PR approved by ≥ 1 reviewer | GitHub PR review approval |

Gate fails → fix the specific failing criterion, do not merge until all pass.

### MODIFIED: `.claude/skills/build/SKILL.md`

Three additions, appended as new sections:

**Addition 1 — Reference to handoff templates** (add to prompt template section, Step 3):
A note after the existing prompt templates stating that agents should use the formats in `docs/workflow/handoff-templates.md` when reporting results — Template 1 for context, Template 2/3 for QA verdicts.

**Addition 2 — Strict mode section** (new `## Strict mode: /build --strict`):
- When to use: auth/payments/critical path features; prior build had cascading failures
- How it works: per-task sequential loop — spawn agent for task N → spawn qa-runner → PASS advance / FAIL retry (max 3) → 3 failures emit Escalation template → pause for human decision
- Output format: each task emits one of Template 2, 3, or 4 from handoff-templates.md
- Note: default parallel mode is unchanged; `--strict` is permanently opt-in

**Addition 3 — Escalation protocol section** (new `## When a task is blocked`):
- Step 1: Check dependency ordering — reorder if a prerequisite is unfinished
- Step 2: Retry with more context — re-read PLAN.md + SPEC.md, clarify acceptance criteria in prompt
- Step 3: Emit Template 4 (Escalation) from handoff-templates.md — document failure history + root cause
- Step 4: Annotate task.md as `[blocked: reason]`, continue unblocked tasks
- Final rule: a phase with blocked tasks cannot pass its exit gate; human decision required
